# Simulation (NVIDIA Isaac Lab)

A GPU physics simulation and reinforcement-learning environment for BugBot, built on
**NVIDIA Isaac Lab** (v2.3.2). It reproduces the *hard* part of BugBot: the chaotic X-omni
drive, and trains a policy that uses sensor feedback to hold a commanded holonomic motion.

> Status: first authored pass. The BugBot-specific logic (the X-omni mix, chaos model,
> observations, reward) is unit-tested via `bugbot_tasks/mixing.py`, but the Isaac Lab env
> has not yet been run on a live install. Expect a validation/tuning pass (search the code
> for `VALIDATE`). It targets the same Isaac venv as the Yertle project
> (`C:\Users\Jerome\isaac\.venv`, Python 3.11).

## Layout

```
simulation/
├── robot/
│   └── bugbot.urdf              chassis + 4 wheels in the X layout (continuous joints)
│   └── usd/bugbot.usd           generated by convert_urdf (step 1 below); not committed yet
├── bugbot_tasks/                importable Isaac Lab task package
│   ├── mixing.py                X-omni mix (matches firmware MotionLib.cpp) -- pure NumPy
│   ├── bugbot_cfg.py            ArticulationCfg (spawns the USD, wheel actuators)
│   ├── bugbot_env.py            BugBotEnv(DirectRLEnv): drive + chaos + sensors + reward
│   ├── bugbot_env_cfg.py        DirectRLEnvCfg: sim/scene/spaces + task, chaos, reward params
│   ├── rsl_rl_ppo_cfg.py        PPO runner config (rsl_rl)
│   ├── train.py                 train on GPU
│   ├── play.py                  roll out a trained policy
│   └── __init__.py              registers the Gym task ids
└── tests/
    └── test_mixing.py           pure-python mix tests (no Isaac needed)
```

## The task

The policy outputs four wheel commands (BR, FR, BL, FL). The env turns them into an
*actual* body twist through the firmware mix **plus a per-env chaos model** (per-wheel
slip/gain, translation<->rotation coupling, vibration, first-order latency), then writes
that motion to the robot. The policy observes the commanded twist and the noisy measured
motion (IMU-like yaw rate + optical-flow-like linear velocity) and must learn the micro
corrections that keep BugBot tracking the command. Reward = velocity-tracking minus effort
and action-rate penalties. Chaos parameters are re-randomised every episode (domain
randomisation is the point, not an afterthought).

## Run it

Verify the drive mix with no Isaac needed:

```bash
cd simulation
python -m pytest tests            # or: PYTHONPATH=. python tests/test_mixing.py
```

**1. Convert the robot model (URDF -> USD)** (once):

```bat
set OMNI_KIT_ACCEPT_EULA=YES
C:\Users\Jerome\isaac\.venv\Scripts\python.exe C:\Users\Jerome\isaac\IsaacLab\scripts\tools\convert_urdf.py ^
    robot\bugbot.urdf robot\usd\bugbot.usd --headless
```

**2. Train (GPU):**

```bat
cd simulation
set OMNI_KIT_ACCEPT_EULA=YES
C:\Users\Jerome\isaac\.venv\Scripts\python.exe -m bugbot_tasks.train --headless --num_envs 4096 --max_iterations 300
```

Logs/checkpoints: `bugbot_tasks/runs/bugbot_track/<timestamp>/` (watch with `tensorboard --logdir bugbot_tasks/runs`).

**3. Play / view:**

```bat
C:\Users\Jerome\isaac\.venv\Scripts\python.exe -m bugbot_tasks.play --checkpoint bugbot_tasks\runs\bugbot_track\<ts>\model_<n>.pt --num_envs 32
```

## Notes / next

- The drive is applied at the **body level** (root velocity), not via omni-roller contact,
  which keeps thousands of envs fast and puts the modelling effort into the chaos. A
  higher-fidelity roller-contact wheel model is a possible follow-up.
- Chassis dimensions, masses, and drive limits in `bugbot_cfg.py` / `bugbot_env_cfg.py` are
  starting points; refine against the mechanical CAD and a real BugBot once available.
- The trained policy is intended to deploy to the real robot via [`../ros2/`](../ros2),
  reusing the same mix.
